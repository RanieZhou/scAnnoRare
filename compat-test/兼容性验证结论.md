# 公网 HTTPS Web 访问 localhost Agent 兼容性验证结论

> 对应设计文档第 17 章「公网 HTTPS Web 访问 localhost Agent 的兼容性验证」P0 项。
> 验证日期：2026-06。

## 验证目标

模拟真实部署场景：Web 平台部署在公网 HTTPS 域名，Local Agent 运行在用户本机
`http://127.0.0.1:17890`。验证浏览器是否允许 **HTTPS 页面访问 HTTP loopback Agent**。

## 验证方法

- 自签名证书将测试页部署在局域网 IP `https://10.112.89.52:8443`（HTTPS、private 网络源），
  复现「HTTPS 页面 → HTTP loopback」的 Mixed Content + 跨域 + Private Network Access 场景。
- 测试页用 `fetch()` 发起三类请求：简单 GET、带 Authorization 头的 GET（触发 CORS 预检 + PNA）、
  带 JSON body 的 POST（触发预检）。
- 用 Playwright 三引擎实测：Chromium（Chrome/Edge）、Firefox、WebKit（Safari）。

## 核心结论

| 浏览器引擎 | HTTPS 页面 → HTTP localhost Agent | 结果 |
|---|---|---|
| **Chromium（Chrome / Edge）** | 3/3 请求成功到达 Agent | ✅ 可用 |
| **Firefox（火狐）** | 3/3 请求成功到达 Agent | ✅ 可用 |
| **WebKit（Safari）** | 0/3，全部 `Not allowed to request resource` | ❌ **被拦截** |

### 根因：Mixed Content（混合内容拦截）

对照实验确认根因：
- WebKit 在 **HTTP 页面** → HTTP loopback Agent：**3/3 通过**；
- WebKit 在 **HTTPS 页面** → HTTP loopback Agent：**0/3 被拦截**。

说明 **Chrome 与 Firefox 对 `localhost / 127.0.0.1` 做了 mixed-content 豁免**，
允许 HTTPS 页面访问本地 HTTP 服务；而 **Safari/WebKit 不豁免 localhost**，
HTTPS 页面无法访问任何 HTTP 资源（含 loopback）。

### Agent 服务端（已就绪）

Agent 的 CORS 与 PNA 预检响应头均正确：
- `access-control-allow-origin`（回显请求 Origin）✅
- `access-control-allow-private-network: true`（PNA 预检）✅
- `access-control-allow-credentials: true` ✅

即问题**不在 Agent 服务端**，而在浏览器对 HTTPS→HTTP 的拦截策略差异。

## 影响与建议

按当前「公网 HTTPS Web + localhost HTTP Agent」架构：
- **Chrome / Edge / Firefox 用户可正常使用**；
- **Safari 用户无法使用**（Mac 默认浏览器，影响面较大）。

可选方案：

| 方案 | 说明 | 代价 |
|---|---|---|
| A. 限定浏览器 | 提示 Safari 用户改用 Chrome/Edge/Firefox | 最低，但牺牲 Safari 体验 |
| B. Agent 提供 HTTPS | Agent 监听 HTTPS + 受信任本地证书，消除 mixed content | 本地可信证书分发复杂 |
| C. 桌面应用内置 WebView | 桌面客户端用 WebView 直接加载 Web，规避跨源 HTTPS→HTTP | 需打包桌面应用，体积大 |
| D. Agent 反向连云端 | Agent 主动通过 WebSocket 连云端拉取任务，不由浏览器直连 | 架构改动较大 |

**建议**：将本结论作为桌面应用形态的决策依据。若必须覆盖 Safari，
优先评估方案 C（桌面应用内置 WebView，与设计文档 17.4 降级方案 A 一致），
其次方案 B。若可接受「仅支持 Chrome/Edge/Firefox」，则维持当前架构，
桌面应用做轻托盘壳即可。

